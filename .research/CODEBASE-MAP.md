# Codebase Map: gsd-pi UAT Skill Project Context

## Project Goal

Build a **UAT-Skill for gsd-pi (GSD v2)** that automates the execution of UAT test cases generated by the existing GSD workflow. GSD v2 already generates `S##-UAT.md` files per slice and has a built-in `run-uat` dispatch unit тАФ but there is **no equivalent of GSD v1's `/gsd:verify-work`** as a dedicated, reusable skill. The built-in `run-uat` is a prompt-based approach that relies on the LLM to execute checks. This skill adds structured, repeatable browser-based UAT execution.

## Source Material: norcalcoop's GSD v1 Skill (Issue #764)

From `gsd-build/get-shit-done` Issue #764 тАФ a community-contributed skill called `gsd-uat-browser` that:

- Reads GSD v1 `{phase}-UAT.md` files
- Translates human test items into YAML spec entries
- Uses `portless` (stable localhost URLs) + `agent-browser` CLI (headless Chromium) to execute
- Has a **Discovery mode** to crawl the codebase and design e2e user journeys
- Handles auth flow detection, credential management, session state
- Produces structured pass/fail reports with screenshots on failure

### Skill System

**Directories:**
- Global (user): `~/.gsd/agent/skills/{skill-name}/`
- Project-local: `.pi/agent/skills/{skill-name}/`

**Format:** GSD v2 skills use `SKILL.md` with:
- YAML frontmatter: `name`, `description`
- **Pure XML body structure** (no markdown headings тАФ use `<objective>`, `<process>`, etc.)
- Router pattern for complex skills: `SKILL.md` тЖТ `workflows/` тЖТ `references/` тЖТ `templates/` тЖТ `scripts/`
- Progressive disclosure: SKILL.md < 500 lines, details in reference files
- Reference files also use pure XML structure

**Discovery:** Skills auto-discovered at session start. Names + descriptions injected into system prompt as `<available_skills>`. Auto-mode detects new skills at unit boundaries.

### Existing UAT System in GSD v2

**Flow:**
1. Execute-task completes a slice тЖТ writes `S##-UAT.md` (using `templates/uat.md`)
2. `auto-dispatch.ts` rule "run-uat (post-completion)" checks `checkNeedsRunUat()`
3. If UAT file exists and no result yet тЖТ dispatches `run-uat` unit
4. `auto-prompts.ts` builds the prompt using `prompts/run-uat.md`
5. LLM executes checks based on UAT type and writes `S##-UAT-RESULT.md`

**UAT Types (from `run-uat.md` prompt):**
- `artifact-driven` тАФ verify via shell commands, file reads, artifact checks
- `browser-executable` тАФ use browser tools to verify UI behavior
- `runtime-executable` тАФ execute scripts, check exit codes
- `live-runtime` тАФ exercise real runtime paths
- `mixed` тАФ combination of above
- `human-experience` тАФ gather evidence, mark subjective checks as `NEEDS-HUMAN`

**UAT Result Format:**
```markdown
---
sliceId: S01
uatType: artifact-driven
verdict: PASS | FAIL | PARTIAL
date: <ISO timestamp>
---
# UAT Result тАФ S01
## Checks
| Check | Mode | Result | Notes |
## Overall Verdict
## Notes
```

**Preference gating:** `prefs.uat_dispatch` must be enabled for auto-dispatch.

**Known bugs (open issues):**
- #1693: Manual `/gsd dispatch uat` targets wrong slice (activeSlice vs last completed)
- #1695: Status UI conflates UAT target with next active slice

### Browser Tools Available in gsd-pi

gsd-pi has extensive built-in browser tools (Playwright-based):
- `browser_navigate`, `browser_click`, `browser_type`, `browser_screenshot`
- `browser_find`, `browser_get_accessibility_tree`, `browser_assert`
- `browser_batch` (multi-step sequences)
- `browser_evaluate` (JS execution)
- `browser_wait_for` (conditions)
- `browser_extract` (structured data)
- `browser_fill_form`, `browser_analyze_form`
- `browser_mock_route`, `browser_block_urls`
- `bg_shell` (background processes тАФ servers, builds)

**Key difference from norcalcoop:** gsd-pi does NOT use `agent-browser` CLI or `portless`. It has native browser tools via Playwright. The skill should use these native tools instead.

### Key Differences: GSD v1 тЖТ GSD v2

| Aspect | GSD v1 (get-shit-done) | GSD v2 (gsd-pi) |
|--------|----------------------|------------------|
| Runtime | Claude Code extension | Standalone TUI (pi) |
| Skill location | `~/.claude/skills/` | `~/.gsd/agent/skills/` or `.pi/agent/skills/` |
| Skill format | Markdown with `#` headings | Pure XML body (no `#` headings) |

| Workflow phases | Numbered phases (01, 02...) | Milestones тЖТ Slices тЖТ Tasks |

## Adaptation Strategy

### What to Keep from norcalcoop

1. **UAT file parsing logic** тАФ reading test items from markdown
2. **Test classification** тАФ mapping UAT items to test types (auth, content, navigation, API, etc.)
3. **Auth discovery flow** тАФ detecting protected routes, handling credentials
4. **Discovery mode** тАФ crawling codebase to design user journeys (valuable addition)
5. **Semantic selectors over CSS** тАФ using labels, roles, text content
6. **Report format** тАФ structured pass/fail with evidence

### What to Replace

1. **`agent-browser` CLI тЖТ native browser tools** (`browser_navigate`, `browser_click`, `browser_find`, `browser_assert`, `browser_batch`)
2. **`portless` тЖТ `bg_shell`** (start dev server, detect readiness via port)
3. **Python scripts тЖТ skill instructions** (gsd-pi skills are prompt-driven, not script-driven; though scripts/ dir is supported)
4. **YAML spec format тЖТ direct browser tool calls** (the LLM can drive browser tools directly rather than generating an intermediate YAML spec)
5. **GSD v1 file paths тЖТ GSD v2 paths** (`.planning/phases/` тЖТ `.gsd/milestones/`)
6. **Markdown headings тЖТ XML tags** in SKILL.md

### Integration Points with GSD v2

1. **UAT file location:** `resolveSliceFile(base, mid, sid, "UAT")` тЖТ `.gsd/milestones/{MID}/slices/{SID}/{SID}-UAT.md`
2. **UAT result location:** `.gsd/milestones/{MID}/slices/{SID}/{SID}-UAT-RESULT.md`
3. **UAT template:** `src/resources/extensions/gsd/templates/uat.md`
4. **Run-UAT prompt:** `src/resources/extensions/gsd/prompts/run-uat.md`
5. **Dispatch rule:** `auto-dispatch.ts` rule "run-uat (post-completion)"
6. **Preference:** `prefs.uat_dispatch` gates auto-dispatch
7. **State:** `checkNeedsRunUat()` in `auto-prompts.ts` determines when to run
