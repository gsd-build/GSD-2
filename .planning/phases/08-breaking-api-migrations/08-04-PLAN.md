import { AgentSessionRuntime, createAgentSessionRuntime, type CreateAgentSessionRuntimeFactory } from "@gsd/pi-coding-agent";

**Step 2 — Add private _runtime field to AgentSession:**
Add a private field `private _runtime: AgentSessionRuntime | undefined;` to the class. It starts undefined and is initialized when the first `createAgentSessionRuntime` call is made during session setup.

**Step 3 — Create a factory closure:**
Create a private method `private _createRuntimeFactory(): CreateAgentSessionRuntimeFactory` that returns a factory function matching the `CreateAgentSessionRuntimeFactory` type. The factory should use `AgentSession`'s existing setup logic (the code that currently runs in the constructor/init path) to create a new `CreateAgentSessionRuntimeResult`.

This is Claude's discretion per CONTEXT.md. The factory closure should capture `this` (the AgentSession) and reuse the existing session setup code path.

**Step 4 — Create state snapshot/restore helpers:**
Create two private methods:
```typescript
private _snapshotState(): { listeners: Map<string, Function[]>; steering: unknown[]; followUp: unknown[]; pending: unknown[] }
private _restoreState(snapshot: ReturnType<typeof this._snapshotState>): void
```

`_snapshotState` captures:
- All event listeners registered via `on()`/`once()` — read how the EventEmitter stores them (likely `this._listeners` or similar internal map)
- `this._steeringMessages` array (shallow copy)
- `this._followUpMessages` array (shallow copy)
- `this._pendingNextTurnMessages` array (shallow copy)

`_restoreState` re-registers all listeners on the new session and restores the message arrays.

**Step 5 — Refactor session transition methods:**
For each of the three transition paths (newSession, switchSession/resume, fork):

1. Call `_snapshotState()` to capture current state
2. Delegate to `_runtime.newSession()` / `_runtime.switchSession()` / `_runtime.fork()`
3. If not cancelled, call `_restoreState(snapshot)` to re-register listeners and restore state arrays
4. The `session_start` emission (migrated in Plan 03) is now handled by AgentSessionRuntime through the factory — per Pitfall 7 in RESEARCH.md, ensure `session_start` is emitted from ONE place only. If AgentSessionRuntime's factory causes `session_start` to be emitted via `createAgentSession`, REMOVE the direct `extensionRunner.emit(session_start)` calls added in Plan 03 from AgentSession's transition methods (since AgentSessionRuntime handles it).

**Step 6 — Preserve public API (D-13):**
The public method signatures called by `gsd-agent-modes` must remain unchanged. The refactor is internal — callers should not need any changes. Verify by checking imports of `AgentSession` in `packages/gsd-agent-modes/src/`:
```bash

```

**CRITICAL — Pitfall 7:** After the refactor, if `AgentSessionRuntime` passes `sessionStartEvent` through to the factory closure which calls `createAgentSession`, the `session_start` event will be emitted by the inner session creation. Remove the explicit `extensionRunner.emit({ type: "session_start", ... })` calls from AgentSession's transition methods to avoid double-emission. The canonical emission point becomes the factory closure.
